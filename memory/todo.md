## Clarified Items

### 시스템 2페이지 IP 표시
- 시스템 2페이지에서 사용자 PC의 로컬 IP가 직접 드러나면 안 된다.
- 표시 기준은 시험 대상 URL의 host 또는 시험 대상 IP로 통일한다.
- 시스템/플랫폼 모두 같은 표시 규칙을 쓰는지 점검 필요.

### 3/4페이지 완료 상태 계산
- 완료는 모든 하위 시험이 끝났을 때만 성립한다.
- 일부 API/시나리오/분야만 끝난 상태는 계속 `진행중`으로 유지한다.
- 타이머 노출 문제와 완료 상태 계산은 분리해서 다룬다.

### 3페이지/4페이지 상세 스키마 통일
- 4페이지 상세 출력은 3페이지와 같은 키 이름, 값 형식, 빈 값 처리 규칙을 따라야 한다.
- request/response/webhook 표현도 동일 기준을 사용한다.

### 점수 API 매핑
- `finalScore` -> 전체 점수
- `groupResults` -> 분야별 결과 목록
- `groupScore` -> 분야별 점수
- `scenarioResults` -> 시나리오별 결과 목록
- `scenarioScore` -> 시나리오별 점수
- 화면 표시와 데이터 저장 모두 같은 매핑 규칙을 사용한다.

### 점수 UI 용어
- 사용자 노출 문구는 `전체 점수 / 분야별 점수 / 시나리오별 점수`로 고정한다.
- 동일 의미에 서로 다른 용어를 섞지 않는다.

### 상세/원본보기 버튼 정책
- `상세 보기`는 데이터가 있을 때만 노출한다.
- `원본보기 버튼`은 원본 페이로드가 있을 때만 노출한다.
- 빈 상태는 빈 상태 문구로 일관 처리한다.

### 4페이지 상태/색상 정책
- `대기 중`, `진행 중`, `완료`, `오류`는 각각 한 가지 시각 규칙만 갖도록 통일한다.
- 초기 상태, 재시도 상태, 완료 후 상태 전이를 다시 정리할 필요가 있다.
